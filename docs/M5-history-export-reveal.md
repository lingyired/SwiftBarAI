## What this milestone delivers

A small UX follow-up to the M5 generator history sheet that
closes the loop on the **"Export…"** button:

1. The user picks a destination through an `NSSavePanel`.
2. `GeneratorHistoryExporter` writes a
   `MANIFEST.json` to the zip root with
   `appVersion` / `appBuild` / `exportedAt` /
   `entryCount` / `provider`, then shells out to
   `/usr/bin/zip` to bundle the entry's audit log.
3. On success, the exporter activates Finder with the new
   zip pre-selected, and the sheet surfaces an
   **"Exported"** success banner that mentions the reveal.

Three small wins from the user's perspective: they no
longer have to dig the file out of wherever the save panel
put it; support tooling can attribute a shared bundle to
the model + endpoint that produced it; and the round-trip
from "Export…" to "send the file to support" is now a
single click.

## The `NSSavePanel` flow

`GeneratorHistoryExporter.exportEntry(_:store:)` is the
`@MainActor` entry point the M5 history sheet calls. The
flow:

```swift
let sourceDir = store.rootDirectory
    .appendingPathComponent(entry.promptId, isDirectory: true)
guard FileManager.default.fileExists(atPath: sourceDir.path) else {
    return .missingDirectory(reason: sourceDir.path)
}
let panel = NSSavePanel()
panel.nameFieldStringValue = "\(entry.promptId).zip"
panel.canCreateDirectories = true
panel.allowedContentTypes = [.zip]
guard panel.runModal() == .OK, let destination = panel.url else {
    return .cancelled
}
return runZip(
    sourceDirectory: sourceDir,
    destination: destination,
    entry: entry
)
```

The panel's `nameFieldStringValue` is the entry's
`promptId` (a stable hash of the `(request, model)` pair)
plus `.zip`, so the user lands on a name that immediately
identifies the run to both support and to the M5 history
view's filter picker.

## The `MANIFEST.json` zip-root

`runZip(sourceDirectory:destination:entry:)` calls
`writeManifest(into:entryCount:provider:)` before
invoking the `zip` binary. The manifest is written into
`sourceDirectory` (the cwd of the `zip -r .` invocation)
so the file lands at the archive root, alongside the
entry's `request.txt` / `response.json` / `menu.json`.

The shape is a small `HistoryExportManifest: Codable` value
type with five fields:

| Field        | Source                                                            |
| ------------ | ----------------------------------------------------------------- |
| `appVersion` | `CFBundleShortVersionString` from `Bundle.main.infoDictionary`, or `"unknown"`. |
| `appBuild`   | `CFBundleVersion` from `Bundle.main.infoDictionary`, or `"unknown"`.            |
| `exportedAt` | `Date()` at export time, encoded as ISO-8601.                     |
| `entryCount` | `1` for the single-entry export path.                             |
| `provider`   | `"remote:<endpointHost>"`, `"local"`, or `"unknown"` (see below). |

Pretty-printed, sorted-key JSON with
`dateEncodingStrategy = .iso8601` — matching the rest of
the project's JSON output.

The `provider` string is derived from
`AIGeneratorHistoryEntry.endpointHost` / `model`:

- `endpointHost` non-empty → `"remote:<host>"`
- `endpointHost` nil + `model` non-empty → `"local"`
- both empty → `"unknown"`

This mirrors the redacted host information the M5 history
sheet already shows in the detail pane's *"Generated by
`<model>` at `<host>`"* section, so a shared zip is
self-describing without cracking open `response.json`.

The current shape, pretty-printed:

```json
{
  "appBuild": "170",
  "appVersion": "1.2.3",
  "entryCount": 1,
  "exportedAt": "2026-06-13T12:34:56Z",
  "provider": "remote:api.openai.com"
}
```

## The `revealInFinder(_:)` step

`runZip(...)` calls a new private
`static func revealInFinder(_ url: URL)` on the success
branch, immediately before
`return .success(destination: destination)`. A failed
zip write still returns
`.zipFailed(reason:)` / `.launchFailed(reason:)` without
ever popping Finder.

```swift
private static func revealInFinder(_ url: URL) {
    os_log(
        "runZip: revealing exported zip in Finder at %{public}@",
        log: log, type: .info, url.path
    )
    DispatchQueue.main.async {
        NSWorkspace.shared.activateFileViewerSelecting([url])
    }
}
```

`NSWorkspace` is documented to be safe from any thread,
but the project keeps UI work on the main thread for
consistency with the rest of the exporter's callers, and
`GeneratorHistoryExporter` itself is not
`@MainActor`-isolated. The reveal is **fire-and-forget**:
a failure to launch Finder (e.g. the user has never opened
Finder) does not surface as an error to the caller — the
zip is already on disk, so the export is still a success.

## The user-facing success banner

`GeneratorHistorySheet.exportEntry(_:)` maps the
exporter's `Result` to a single `NSAlert`:

```swift
case .success(let destination):
    showAlert(
        title: "Exported",
        message: "Exported to \(destination.lastPathComponent). Finder has been opened to the file."
    )
case .cancelled:
    break
case .missingDirectory(let reason):
    showAlert(
        title: "Export failed",
        message: "The entry \"\(entry.promptId)\" has no on-disk directory at \(reason)."
    )
case .zipFailed(let reason), .launchFailed(let reason):
    showAlert(title: "Export failed", message: reason)
```

The success message uses `lastPathComponent` (the file
name) rather than the full path because the alert already
says where the file is — "Finder has been opened to the
file" — and a full path in the same line is noise. The
alert style is `.informational` for success and
`.warning` for any failure.

## Tests

`menubar01Tests/GeneratorHistoryExporterTests.swift` has
5 Swift Testing tests pinning down the export path:

- `testExport_writesNonEmptyZipAndExitsZero` —
  pre-existing, asserts the happy-path round-trip writes
  a non-empty zip.
- `testRunZip_succeeds_doesNotThrow` — sanity counterpart
  asserting the happy-path source + destination round-trip
  returns `.success(destination:)`. The new
  `revealInFinder(_:)` call lives on this code path, so
  a regression that broke the success branch would also
  break the UX follow-up.
- `testExportEntry_zipContainsManifest` — exports an
  entry with `endpointHost: "api.openai.com"`, unzips,
  asserts `MANIFEST.json` exists at the archive root.
- `testExportEntry_manifestHasCorrectProvider` — same
  entry, asserts the manifest's `provider` field is
  `"remote:api.openai.com"`.
- `testExportEntry_manifestProviderIsLocalForNilHost` —
  `endpointHost: nil`, `model: "local-model"`, asserts
  `provider == "local"`.
- `testExportEntry_manifestHasCorrectEntryCount` —
  single entry, asserts `entryCount == 1`.

The `testRunZip_succeeds_doesNotThrow` test's docstring
documents the deliberate omission of an
`NSWorkspace.shared.activateFileViewerSelecting` assertion:
`NSWorkspace` has no test seam (system singleton, no
injection point in `GeneratorHistoryExporter`), and adding
a protocol abstraction for a single one-liner would be
over-engineering. The visible behaviour (Finder pops up)
is a UX nicety, not a correctness requirement.

## Out of scope (follow-ups)

- **Multi-entry exports.** The M5 history sheet exports
  one entry at a time. A future "Export All…" affordance
  would set `entryCount` to the actual number of entries
  in the bundle.
- **Reveal on cancel.** A user who clicks "Export…", picks
  a destination, then changes their mind and clicks
  Cancel in a future "are you sure?" prompt has no
  current path to a Finder reveal — the existing
  `NSSavePanel.runModal() == .OK` short-circuit returns
  `.cancelled` before `runZip` runs. Tracked but not
  blocking.
- **iCloud / sandbox-aware destinations.** The MAS build
  runs in a sandbox and may have different destination
  paths visible in Finder. A follow-up can route the
  `revealInFinder(_:)` call through the security-scoped
  resource API the MAS entitlements already provide.
