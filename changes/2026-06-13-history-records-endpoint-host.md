# 2026-06-13: AI generator history records endpoint host

- **Type:** feat
- **Scope:**
  `menubar01/AI/AIGeneratorHistoryEntry.swift`,
  `menubar01/AI/AIGenerator.swift`,
  `menubar01/AI/RemoteAIPluginGenerator.swift`,
  `menubar01/UI/Plugin Generator/AIGeneratorViewModel.swift`,
  `menubar01Tests/AIGeneratorEndpointHostTests.swift`
- **Author(s):** Trae AI
- **Commit(s):** TBD
- **Status:** in-progress

## Summary

Closes the M5 history-UI follow-up noted at the end of
[`2026-06-13-remote-ai-plugin-generator.md`](2026-06-13-remote-ai-plugin-generator.md):
`AIGeneratorHistoryEntry` now records the endpoint host (for
remote runs) so the history UI can render
"Generated by `<model>` at `<host>`" alongside each entry
without leaking the full URL or the apiKey. The
`AIPluginGenerator` protocol gains a new `var endpointHost:
String?` requirement (defaulted to `nil`) and
`RemoteAIPluginGenerator` overrides it with
`endpoint.host`. The view model's `generate()` records the
host on every successful `AIGeneratorHistoryEntry`.

## Motivation

The M5 history sheet already shows the model name
(`entry.model`) and the user's request, but remote runs were
visually indistinguishable from local / mock runs because the
generator's endpoint was not persisted on the history entry.
The M5 follow-up bullet in the remote-generator change record
called this out explicitly:

> AIGeneratorHistoryStore should be updated to record
> `endpoint.host` and `model` alongside the `promptId`, so
> the M5 history UI can show "Generated by gpt-4o-mini at
> api.openai.com" without leaking the full URL or the
> apiKey.

Storing only the host (not the full URL, not the apiKey) keeps
the on-disk history safe to share via the future
"Export… → .zip" flow (which already pulls
`request.txt` + `response.json` + `menu.json` into an audit
bundle).

## Changes

### `menubar01/AI/AIGeneratorHistoryEntry.swift`

- New `public let endpointHost: String?` field, defaulted to
  `nil` for backward compatibility with any `response.json`
  files written before this change.
- `init(...)` gains a trailing
  `endpointHost: String? = nil` parameter so the M5 v1 callers
  that omit it keep compiling unchanged.
- `Codable` updated:
  - New `case endpointHost` in the hand-rolled `CodingKeys`
    enum.
  - `init(from:)` decodes via `decodeIfPresent(_:forKey:)` so
    a v1 `response.json` that predates the field decodes to
    `nil` rather than throwing.
  - `encode(to:)` uses `encodeIfPresent(_:forKey:)` so the
    field is omitted from the on-disk JSON when `nil`,
    matching the existing `menuTreeJSON` convention.
- `==` hand-rolled comparison now includes
  `endpointHost` in the guard.

### `menubar01/AI/AIGenerator.swift`

- `AIPluginGenerator` protocol gains a new
  `var endpointHost: String? { get }` requirement, documented
  as "host portion of the endpoint that produced `plugin`,
  or `nil` for providers that do not dial out".
- A `public extension AIPluginGenerator` block provides a
  `var endpointHost: String? { nil }` default so Mock,
  Local, LocalEcho, RemoteEcho, and any future provider do
  not have to write the override.

### `menubar01/AI/RemoteAIPluginGenerator.swift`

- New `public var endpointHost: String? { endpoint.host }`
  override on `RemoteAIPluginGenerator`. Returns
  `URL.host`'s native `String?` (which is `nil` for
  malformed endpoints), matching the protocol's signature.

### `menubar01/UI/Plugin Generator/AIGeneratorViewModel.swift`

- The `historyStore.record(AIGeneratorHistoryEntry(...))`
  call inside `generate()` now passes
  `endpointHost: generator.endpointHost`. The error-handling
  / non-fatal log line is unchanged.

### `menubar01Tests/AIGeneratorEndpointHostTests.swift` (new)

- 5 Swift-Testing tests across 2 suites:
  - `AIGeneratorHistoryEntryEndpointHostTests`:
    - `testEntry_encodesAndDecodesEndpointHost` —
      round-trips an entry whose `endpointHost` is
      `"api.openai.com"` through the
      `FileSystemAIGeneratorHistoryStore.defaultEncoder()`
      so the JSON assertion matches the on-disk format
      (pretty-printed, sorted keys, ISO-8601 dates), and
      asserts `decoded.endpointHost == "api.openai.com"`.
    - `testEntry_decodesMissingEndpointHostAsNil` —
      round-trips an entry constructed with
      `endpointHost: nil` (the default) and asserts the
      on-disk JSON omits the key and decoding leaves the
      field at `nil`.
    - `testEntry_decodesLegacyJSONWithoutEndpointHostKey` —
      hand-crafts a v1-shaped JSON blob (Unix-seconds
      `createdAt`, no `endpointHost` key) and asserts
      decoding succeeds with `endpointHost == nil`. This is
      the backward-compat test for existing on-disk
      `response.json` files.
  - `AIPluginGeneratorEndpointHostTests`:
    - `testRemoteAIGenerator_exposesEndpointHost` —
      constructs a `RemoteAIPluginGenerator` with
      `https://api.openai.com` and asserts
      `generator.endpointHost == "api.openai.com"`.
    - `testMockGenerator_endpointHostIsNil` — constructs a
      `MockAIPluginGenerator` and asserts
      `generator.endpointHost == nil`.

## Backward compatibility

- v1 `response.json` files (no `endpointHost` key) decode
  cleanly: the hand-rolled `init(from:)` uses
  `decodeIfPresent(_:forKey:)`.
- New `response.json` files omit the `endpointHost` key
  when the value is `nil`, so the on-disk shape stays
  minimal for local / mock runs.
- `init(...)` defaults the new parameter to `nil`, so
  existing call sites that build a history entry
  programmatically (e.g. unit tests in
  `AIGeneratorHistoryStoreTests`) keep compiling unchanged.

## Security

- The full URL and the apiKey are never persisted on the
  history entry. Only `URL.host` is exposed through the new
  protocol accessor, and the new entry field carries just
  that string.
- This mirrors the redacted `os_log` line in
  `RemoteAIPluginGenerator.init`, which already shows only
  the host (and a redacted apiKey) for diagnostics.
